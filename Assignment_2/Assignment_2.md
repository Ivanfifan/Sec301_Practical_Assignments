# Practical Assignment 2 — Semenchenko Ivan

## Опис бази даних

Ця база даних — спрощена модель e-commerce сервісу з трьома таблицями:

| Таблиця | Опис |
|---|---|
| `opt_clients` | Профілі клієнтів з іменем, email та статусом підписки (active/inactive) |
| `opt_orders` | Замовлення клієнтів з датою та посиланням на продукт |
| `opt_products` | Каталог продуктів з назвою та категорією |

## Функціонал та результати запиту

Запит аналізує активність клієнтів: фільтрує лише тих, хто має статус `active`, враховує тільки замовлення з категорій `Category1` та `Category3`, групує замовлення по кожному клієнту через `GROUP BY` і порівнює їх кількість із середнім значенням, присвоюючи мітку через `CASE WHEN`.

Результат виводиться у форматі:

| name | surname | email | total_orders | activity_level |
|---|---|---|---|---|

Де `activity_level` може бути `Above Average`, `Below Average` або `Average`.

## Пояснення запиту

### Неоптимізований варіант

Початковий запит складається з трьох `SELECT`, об'єднаних через `UNION ALL` — окремо для `Above Average`, `Below Average` та `Average`. Кожна гілка самостійно джойнить всі три таблиці і окремо обчислює підзапит з `AVG` через вкладений `SELECT`. Тобто одна й та сама робота виконується тричі — три рази джойни, шість разів обчислення середнього.

### Оптимізований варіант

Замість `UNION ALL` використано CTE (`WITH subquery`), яка обчислює дані один раз — джойнить таблиці, групує, рахує `total_orders`. Зовнішній `SELECT` читає з CTE і визначає категорію через `CASE WHEN` без жодних повторних джойнів.

### Індексація

Додано covering index на `opt_products`:

```sql
CREATE INDEX idx_products_category_cover ON opt_products(product_category)
INCLUDE (product_id);
```

В плані виконання після додавання індексу з'явився `Index Only Scan` з `Heap Fetches: 0` — база отримує всі потрібні дані прямо з індексу.

## Порівняння планів виконання

### Неоптимізований запит
```
Execution Time: ~1100ms
Seq Scan на всіх таблицях
AVG обчислюється 6 разів (по 2 на кожну гілку UNION ALL)
Таблиці джойняться 3 рази
```

### Оптимізований запит
```
Execution Time: ~400ms
Index Only Scan на opt_products (Heap Fetches: 0)
AVG обчислюється 2 рази
Таблиці джойняться 1 раз
```

| | Неоптимізований | Оптимізований |
|---|---|---|
| Проходів по таблицях | 3× | 1× |
| AVG обчислень | 6 | 2 |
| Scan на opt_products | Seq Scan | Index Only Scan |
| Час виконання | ~1100ms | ~400ms |

## Управління планувальником (Planner Settings)

Для демонстрації впливу налаштувань на план виконання змінено параметр `work_mem`:

```sql
SET work_mem = '256MB';
```

До змін сортування і CTE скидались на диск:

```
Sort Method: external merge  Disk: 9848kB
CTE Storage: Disk  Maximum Storage: 7145kB
Execution Time: 398ms
```

Після змін CTE перейшов в RAM, однак планувальник відмовився від паралельного виконання і перейшов на `HashAggregate` в один потік:

```
Sort Method: quicksort  Memory
CTE Storage: Memory  Maximum Storage: 9512kB
Execution Time: 650ms
```

Це показує, що planner settings впливають не лише на конкретну операцію, а на весь план цілком — планувальник динамічно перебудовує стратегію залежно від доступних ресурсів, і зміни не завжди дають очікуваний результат.
